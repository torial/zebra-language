# Debugging Zebra programs

Zebra ships a DAP (Debug Adapter Protocol) proxy in the `zebra debug` subcommand.
It sits between your IDE and `lldb-dap`, remapping source locations between the
`.zbr` source you write and the `.zig` file generated by the compiler.

```
IDE ─── DAP ──→ zebra debug ─── DAP ──→ lldb-dap ─── DWARF ──→ binary
                      ↑
              source-map remapping
              zbr ↔ zig lines
```

---

## Prerequisites

`lldb-dap` (or the legacy name `lldb-vscode`) must be on `PATH`.

| Platform | Install |
|----------|---------|
| Windows  | `winget install LLVM.LLVM` — then add `C:\Program Files\LLVM\bin` to PATH |
| Ubuntu   | `sudo apt install lldb` |
| macOS    | `brew install llvm` or `xcode-select --install` |

Verify: `lldb-dap --version` (or `lldb-vscode --version`).

---

## Command-line usage

```bash
zebra debug file.zbr
```

This:
1. Compiles `file.zbr` → `file.zig` with source-map comments.
2. Runs `zig build-exe file.zig -O Debug -lc` to produce a debug binary.
3. Spawns `lldb-dap` as a child process.
4. Relays DAP messages over **stdin/stdout**, remapping source locations.

The proxy speaks standard DAP over stdin/stdout. Any IDE that can launch a debug
adapter as a subprocess and communicate via stdio can use this directly.

### Listen mode (custom IDE / ZebraIDE)

```bash
zebra debug --listen 54321 file.zbr
```

Same compilation steps, but instead of using stdin/stdout the proxy:
1. Binds TCP port 54321 and prints:
   ```
   zebra debug: listening for IDE on 127.0.0.1:54321...
   ```
2. Waits for the IDE to connect on that port.
3. Relays DAP over the TCP stream.

This is the mode ZebraIDE (and any custom IDE) should use. See the
[ZebraIDE](#zebra-ide) section below.

## VS Code

### Option A — CodeLLDB workaround (works today, no extension required)

This approach compiles the program first, then uses the
[CodeLLDB](https://marketplace.visualstudio.com/items?itemName=vadimcn.vscode-lldb)
extension to debug the resulting binary. You get full LLDB debugging; breakpoints
show `.zig` file lines rather than `.zbr` lines, but the experience is otherwise
complete.

**Install CodeLLDB** from the VS Code marketplace.

Add to `.vscode/tasks.json`:

```json
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "zebra: compile debug",
      "type": "shell",
      "command": "zebra",
      "args": ["debug", "--compile-only", "${file}"],
      "group": "build",
      "presentation": { "reveal": "silent", "panel": "shared" },
      "problemMatcher": []
    }
  ]
}
```

> **Note:** `--compile-only` mode for `zebra debug` is not yet implemented.
> As a workaround, use a two-command task:
> ```json
> "command": "zebra --emit-zig ${file} && zig build-exe ${fileBasenameNoExtension}.zig -O Debug -lc"
> ```
> The binary will be written to the current working directory.

Add to `.vscode/launch.json`:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "lldb",
      "request": "launch",
      "name": "Debug Zebra (CodeLLDB)",
      "program": "${workspaceFolder}/${fileBasenameNoExtension}",
      "args": [],
      "cwd": "${workspaceFolder}",
      "preLaunchTask": "zebra: compile debug",
      "sourceMap": {
        "${workspaceFolder}": "${workspaceFolder}"
      }
    }
  ]
}
```

On Windows, change `"program"` to `"${workspaceFolder}/${fileBasenameNoExtension}.exe"`.

### Option B — VS Code extension (future, full source mapping)

The proper integration requires a thin VS Code extension that registers `"zebra"`
as a debug adapter type and launches `zebra debug` as the adapter executable.
When the extension is available, the `launch.json` will look like:

```json
{
  "type": "zebra",
  "request": "launch",
  "name": "Debug Zebra",
  "program": "${file}"
}
```

The extension's `package.json` declares:
```json
{
  "debuggers": [{
    "type": "zebra",
    "label": "Zebra",
    "program": "./out/adapter.js",
    "runtime": "node"
  }]
}
```

The adapter executable only needs to shell out to `zebra debug ${file}` and
pipe stdin/stdout through — the zebra proxy does all the real work.

## ZebraIDE

ZebraIDE uses listen mode. The IDE starts the proxy with a randomly chosen port,
waits for the `"listening for IDE"` banner on stderr, then opens a TCP connection
to that port. All subsequent DAP traffic flows over TCP.

The `problemMatcher` pattern for a VS Code background task that waits for the
proxy to be ready:

```json
{
  "label": "zebra debug proxy",
  "type": "shell",
  "command": "zebra debug --listen 54321 ${file}",
  "isBackground": true,
  "problemMatcher": {
    "owner": "zebra",
    "pattern": { "regexp": "." },
    "background": {
      "activeOnStart": true,
      "beginsPattern": "zebra debug:",
      "endsPattern": "listening for IDE"
    }
  }
}
```

## Source map format

The Zebra compiler emits `// zbr:` comment markers in generated Zig:

```zig
// zbr:path/to/file.zbr:42
some_generated_zig_statement();
```

The proxy builds a bidirectional map from these:
- **Forward** (zbr → zig): used in `setBreakpoints` — IDE sends a `.zbr` line,
  proxy rewrites it to the corresponding `.zig` line before forwarding to lldb-dap.
- **Reverse** (zig → zbr): used in `stackTrace` responses — lldb-dap returns `.zig`
  lines, proxy rewrites them back to `.zbr` lines before forwarding to the IDE.

The source key is normalized to forward-slash separators for cross-platform
matching. Path matching uses suffix comparison so relative paths in markers
correctly match absolute IDE-provided paths.

## Troubleshooting

**`lldb-dap not found on PATH`**
Install LLDB (see Prerequisites) and ensure `lldb-dap` or `lldb-vscode` is in PATH.

**`zebra debug: relay read: BrokenPipe`**
Normal — this appears when the IDE disconnects at the end of a session. Not an error.

**Breakpoints not hitting**
1. Confirm the `.zig` file exists next to your `.zbr` file — it's created by the compile step.
2. Check that `lldb-dap` is version 14+ (older versions have DAP compatibility issues).
3. On Windows, `LLDB_DISABLE_PYTHON=1` is set automatically to avoid a crash with mismatched Python ABI — this is intentional.

**Stack trace shows `.zig` file lines instead of `.zbr`**
The source map is per-session. If the `.zig` file was regenerated without restarting
the debug session, the map may be stale. Restart `zebra debug`.
