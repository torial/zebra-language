# Zebra 1.0 freeze checklist

*Drafted 2026-06-24. The concrete, checkable "do these, then flip the switch"
list. Pairs with `ROADMAP_TO_1.0.md` (strategy) and `API_FREEZE_AUDIT.md`
(the stdlib regret analysis). Items marked 🔲 need doing/deciding; ✅ are already
true; 👤 need Sean specifically.*

## A. Stdlib API consistency sweep (from `API_FREEZE_AUDIT.md`)

The freeze is "add but never remove," so these get locked permanently. Tier 1 is
the recommended pre-freeze fix list:

- ✅ **A1 — unify string-sequence returns on `List(str)`.** DONE 2026-06-24.
  `Net.resolve`, `re.findAll`, `re.groups` now return `List(str)` (preamble fns
  return `std.ArrayList([]const u8)`; both compilers route their for-loops through
  `.items`; selfhost gained `isStrListIter`, bootstrap gained a genForIn dispatch
  arm). Verified on both compilers (regex_groups/regex/dns run identically), gate
  byte-identical, smoke 173/173. Side-fix: two `std.ArrayListUnmanaged(u8) = .{}`
  preamble initializers that Zig 0.16 rejects (`→ .empty`); these were invisible
  because the smoke suite only emits, never compiles (see §D note).
- 🔶 **A2 — RESCOPED (2026-06-24, needs Sean's call).** On inspection two of the
  three audited items were wrong: `Mime.fromExt` returns a sensible default
  (`application/octet-stream`), not a sentinel — leave it; and there is no `Csv.field`
  method (zero callers) — nothing to change. The real item is the **`File` error
  surface**: `modtime`/`size` return `-1`, `read` panics. Recommend one consistent
  sweep making `File.read`/`size`/`modtime` all `throws` (Sean ratified throws for
  modtime). DECISION NEEDED: do the whole-File sweep, or just modtime (0 callers)?
- 🔲 **A3 — `Random` instance form** (`Random.new(seed)` + `rng.int/float/bool/
  bytes`) as the explicit primary API. The thread-local statics already shipped
  (e1f1e1c) — this is the additive instance surface.
- 👤 **B1–B4 — decisions to record** (parser optional-vs-throw consistency;
  whether connection `?` should ever carry a failure reason; inclusive-`randInt`
  and fetch-add semantics documented; `Dir.list` alias). See audit Tier 2.

## D. Real-world validation

- ✅ **Engine + real game logic runs** (2026-06-24): the GameEngine integration
  suite passes headlessly — health/regen/revive, event bus, weighted loot tables
  (genuine `monster_mayhem` mechanics) — confirming Zebra runs a substantial real
  program. (`docs/ROADMAP_TO_1.0.md` runtime frontier, validated.)
- 🔲 **Broaden the runtime pass** — wire more game scripts end-to-end; treat each
  runtime gap as a prioritized bug. (Front-end is 1482/1581 but only measures
  compiles; runtime is where remaining bugs live.) Not strictly a freeze blocker,
  but the strongest evidence for the 1.0 claim.
- 🔲 **Add a compile-check mode to the smoke suite.** Discovered 2026-06-24:
  `tools/selfhost_smoke.sh` only runs `--emit-zig` — it never invokes `zig` on the
  output, so emitted Zig that doesn't compile (e.g. the Zig-0.16 `.{}`→`.empty`
  ArrayList breakage in the regex preamble) passes smoke silently. The byte-identical
  gate has the same blind spot. A `--compile` pass (zig-compile a representative
  subset, or all of `test/`) would close it. High-value stability infra; not a
  freeze blocker but it undermines confidence in "runtime validated" until added.
  (Note: dev default Zig is now **0.16.0** via `zvm use 0.16.0`; 0.17-dev rejects the
  array-repeat `**` and other forms the emitted preamble uses.)

## Draft stability promise (for Sean to edit/ratify)

> **Zebra 1.0 stability commitment.** From 1.0 onward, the public surface — the
> language syntax and semantics, and the signatures of the standard-library
> modules documented in QUICKSTART §31 — is **stable**. Within the 1.x series we
> may *add* (new modules, new methods, new optional/defaulted parameters, new
> syntax that doesn't change the meaning of existing code) but will not *remove*
> or *break* existing surface. Breaking changes are reserved for a future 2.0 and
> will be called out in the CHANGELOG with a migration note.
>
> Explicitly **not** covered by the promise: (a) generated-`.zig` output shape and
> internal compiler APIs; (b) the `zig"…"` escape hatch (its behavior tracks the
> underlying Zig version); (c) anything documented as "experimental" or "preview."
>
> Deprecation policy: a feature slated for removal at 2.0 is marked deprecated in
> a 1.x release (compiler warning) at least one minor version before removal.
